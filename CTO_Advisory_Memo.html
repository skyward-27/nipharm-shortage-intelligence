Overall Verdict: The Core Thesis Is Sound. The Execution Plan Has Five Critical Gaps That Will Cause Expensive Rework If Not Addressed Before a Single Line of Code Is Written.

The "unprofitable floor" insight is genuinely novel and probably correct. The signal architecture is thoughtful. The n8n automation approach is pragmatic. But the PRD treats the ML model as the hard problem — it's not. The hard problems are: (1) you have no validated proof the thesis works on real data, (2) there's no database schema, (3) the recommendation engine is a direction indicator not a decision tool, (4) the competitive moat is being built in the wrong place, and (5) WaveData is being treated as a data source when it's actually an existential dependency. These five gaps must be addressed before Week 4.

01

The Five Critical Gaps

Act Before You Build
1

The Hypothesis Has Never Been Backtested

The "unprofitable floor predicts shortage" claim is attributed to WaveData's 24-year analysis. But has this team run that analysis against the freely available NHSBSA data? Before building a 12-month roadmap on this thesis, spend 2 days in Python validating it. Take every price concession event from NHSBSA records (2019–2024). Ask: was the Drug Tariff price within 10% of its 5-year low in the 6 months prior? If the answer is "yes, in >70% of cases," the thesis is validated. If it's 40%, rethink the entire architecture.

⛔ Blocker — Validate Before Week 4
2

The Molecule Identity Problem Will Break Everything

The platform pulls data from 10+ sources. Each uses a different drug identifier: NHSBSA uses BNF codes, WaveData uses proprietary IDs, DaSH portal uses free-text product names, wholesalers use their own catalogue numbers. Without a clean "Molecule Master" database mapping all identifiers, every signal JOIN is unreliable. You'll conflate Metformin 500mg and 850mg, miss strengths, double-count generics. This is Week 1 work that nobody is talking about — and it will silently corrupt every analysis built on top.

⛔ Blocker — Solve Before Any Data Integration
3

The Recommendation Engine Is a Compass, Not a GPS

"BUY FORWARD" tells a procurement manager the direction. It doesn't tell them how much to buy, by when, at what cost, and what they're risking by waiting. The difference between "BUY FORWARD" and "BUY FORWARD — order 4.5 additional weeks now, cost £12,400, expected saving if concession materialises: £47,000, decision must be made by 15th March" is the difference between a dashboard nobody acts on and a tool that changes procurement behaviour. The financial calculator layer is missing entirely from the PRD.

⚠ Critical Enhancement — Before Phase 1 Launch
4

The Moat Is Being Built in the Wrong Place

The PRD treats the ML model as the competitive moat. It isn't — LightGBM trained on NHSBSA historical data can be replicated by a funded competitor in 3 months. The real moat is the proprietary outcome dataset: which recommendations were acted on, what actually happened, did the price move as predicted? Once you have 18 months of this data, no new entrant can replicate it regardless of how good their model is. The outcome tracking database isn't scheduled until Phase 2. It should be Day 1 infrastructure.

⚠ Strategic Priority Inversion — Move to Phase 1
5

WaveData Is an Existential Dependency Disguised as a Data Source

The PRD lists WaveData as one integration among many. But WaveData's 24-year historical price data is the foundation for the unprofitable floor calculation — the platform's entire core IP. If WaveData costs £80–120K/year, it becomes the single largest cost item. If WaveData launches a competing product, they have everything needed to do so. If WaveData changes API terms, the platform degrades overnight. This needs a mitigation strategy now: build a parallel floor calculator from NHSBSA historical data. Prove it works to 70% accuracy. Then WaveData becomes an enhancement, not a single point of failure.

⛔ Risk — Mitigation Required Before Phase 1
+

The 57-Signal Model Has a Dimensionality Problem

With ~200–400 price concession events historically as training labels, a 57-feature model is severely underdetermined. LightGBM handles sparse data better than most, but the model will learn noise rather than signal. The solution: reduce 57 signals to 8 composite scores through signal clustering. "Supply Stress Score," "Demand Pressure Score," "Market Structure Score," etc. This improves model stability, interpretability, and dramatically reduces the risk of overfitting to the training period. It also makes the explainability layer simpler: "your score is driven by Supply Stress and Demand Pressure."

⚠ Architectural Decision — Before ML Model Design

The Database Schema — The Most Important Architectural Decision You're Not Making

Architecture

The PRD has detailed feature specs, signal descriptions, and a delivery roadmap. It has no database schema. This is the most dangerous omission, because every subsequent engineering decision — signal storage, prediction queries, recommendation generation, outcome tracking — depends on getting the schema right. Here is the schema that should anchor the entire platform:

-- MOLECULE MASTER (the foundation that solves the identity problem)
molecules (
  molecule_id UUID PRIMARY KEY,
  bnf_code VARCHAR(15) UNIQUE NOT NULL, -- Primary identifier, cross-source
  bnf_name TEXT NOT NULL,
  drug_name TEXT,
  form TEXT, -- tablet, capsule, liquid, injection
  strength TEXT,
  wavedata_id VARCHAR(50), -- WaveData proprietary ID mapping
  openprescribing_idVARCHAR(50),
  therapeutic_category TEXT,
  monitoring_tier VARCHAR(10), -- 'watchlist', 'standard', 'low-risk'
  created_at TIMESTAMPTZ DEFAULT NOW()
);
-- PRICE HISTORY (time-series, TimescaleDB hypertable)
price_history (
  molecule_id UUID REFERENCES molecules,
  effective_date DATE NOT NULL,
  tariff_price_penceINTEGER, -- Category A/C/M Drug Tariff price
  concession_price_pence INTEGER, -- NULL if no active concession
  unprofitable_floor_pence INTEGER, -- Calculated floor for this molecule
  distance_from_floor_pct DECIMAL(5,2), -- (tariff_price - floor) / floor * 100
  manufacturer_countSMALLINT, -- From NHSBSA Part VIIIA this month
  data_source VARCHAR(20), -- 'nhsbsa_api', 'nhsbsa_pdf', 'wavedata'
  PRIMARY KEY (molecule_id, effective_date)
);
-- SIGNAL READINGS (every signal, every molecule, every refresh)
signal_readings (
  molecule_id UUID REFERENCES molecules,
  signal_id VARCHAR(50) NOT NULL, -- e.g., 'oos_frequency', 'cpe_requests'
  signal_tier SMALLINT, -- 1, 2, 3, or 4
  composite_score VARCHAR(30), -- Which composite it feeds: 'supply_stress'
  raw_value JSONB, -- Actual signal data (flexible)
  normalised_score DECIMAL(5,2), -- 0.0 to 1.0 (green to red)
  status VARCHAR(10), -- 'green', 'amber', 'red'
  recorded_at TIMESTAMPTZ NOT NULL,
  data_age_hours INTEGER, -- ⚡ KEY: staleness detection
  is_fresh BOOLEAN GENERATED ALWAYS AS ... -- Auto-calculated from max_age config
);
-- PREDICTIONS (model output, every molecule, every run)
predictions (
  prediction_id UUID PRIMARY KEY,
  molecule_id UUID REFERENCES molecules,
  prediction_date DATE NOT NULL,
  horizon_months SMALLINT, -- 1, 3, or 6
  concession_prob DECIMAL(4,3), -- 0.000 to 1.000
  price_direction VARCHAR(20), -- 'strong_up', 'moderate_up', 'stable', etc.
  confidence_pct DECIMAL(4,1),
  model_version VARCHAR(20), -- For retraining comparison
  top_signals_json JSONB, -- Top 3 signals driving this prediction
  outcome VARCHAR(20), -- NULL until horizon passes; 'correct'/'incorrect'
  outcome_date DATE -- When outcome was recorded — THIS IS THE MOAT
);
-- RECOMMENDATIONS + OUTCOMES (the moat table — never Phase 2, always Day 1)
recommendations (
  rec_id UUID PRIMARY KEY,
  molecule_id UUID REFERENCES molecules,
  rec_date DATE NOT NULL,
  action_type VARCHAR(30), -- 'buy_forward', 'buffer_stock', 'hold', etc.
  urgency_score DECIMAL(3,1), -- 0-10
  rationale_text TEXT,
  recommended_weeks_stock DECIMAL(4,1),
  current_stock_weeks DECIMAL(4,1),
  order_units_suggested INTEGER,
  cost_to_act_gbp DECIMAL(10,2), -- Financial calculator output
  expected_saving_gbp DECIMAL(10,2), -- If prediction correct
  decision_deadline DATE, -- "Act by this date to capture the opportunity"
  was_shown BOOLEAN DEFAULT FALSE,
  was_acted_on BOOLEAN, -- NULL = not yet decided
  acted_on_date DATE,
  actual_outcome VARCHAR(20), -- 'correct_acted', 'correct_ignored', 'wrong_acted', 'wrong_ignored'
  pnl_impact_gbp DECIMAL(10,2) -- Actual financial impact — this is the ROI metric
);
⚡
The data_age_hours and is_fresh Columns Are Non-Negotiable

The PRD mentions n8n automation failures as a risk, with the mitigation "show data age warning to users." That's not enough. Every signal must carry a staleness flag, and any signal older than its expected refresh window must display as grey/stale on the dashboard rather than appearing current. A procurement manager seeing a "green" OOS signal that was last updated 3 weeks ago will make a wrong decision with confidence. This is one of the most common failure modes in operational dashboards.

💡
The "Risk-Adjusted Saving" Calculation — How to Build It

Risk-adjusted saving = (expected_saving_if_correct × concession_probability) − (overstocking_cost × (1 − concession_probability)). This requires: (1) historical average price concession uplift per category, (2) current DT price, (3) estimated order quantity from purchasing system. When the risk-adjusted saving is positive, BUY FORWARD. When negative, MONITOR. This single calculation turns the recommendation from subjective guidance into an objective financial decision.

Signal Reduction — From 57 Features to 8 Composite Scores

ML Architecture

Feeding 57 signals into a model with ~400 training examples creates severe overfitting risk. More signals is not a better model — it's a fragile model that memorises training noise. The solution is to cluster correlated signals into 8 independent composite scores. This also makes the explainability layer dramatically simpler: "your risk score is driven primarily by Supply Stress (0.82) and Demand Pressure (0.71)."

📐
How to Calculate Each Composite Score

For each composite score: (1) normalise each constituent signal to 0–1 scale, (2) apply the weighted average using the PRD's existing signal priority weights, (3) output a single 0–1 score with a green/amber/red threshold. The ML model then takes 8 inputs, not 57 — dramatically reducing overfitting, improving training data sufficiency, and making the model retrainable in minutes rather than hours.

The Real Competitive Moat — Building It From Day One

Strategic Architecture

Most SaaS platforms think of their moat as their algorithm. The algorithm is table stakes — it can be replicated. The real moat in a prediction product is the outcome feedback loop: proprietary knowledge of which predictions were correct, under which conditions, and with what financial impact. This data cannot be purchased or replicated.

🔴 Weakest Moat: The Algorithm

LightGBM on NHSBSA data. Replicable by a funded competitor in 3–6 months. This is what the PRD focuses on building.

Replicable: 3–6 months

🟡 Moderate Moat: Historical Signal Data

5+ years of signal readings per molecule. Proprietary aggregated wholesaler OOS data. Requires time to accumulate but data sources are mostly public.

Replicable: 12–18 months

🔵 Strong Moat: Network Effects (Multi-Customer)

Aggregated buying patterns from 10+ pharmacy groups creates demand signal impossible to replicate from public data. Each new customer improves the model for all customers.

Replicable: 24–36 months

🟢 Strongest Moat: Prediction Outcome Database

The recommendations table: which drugs, which action, acted on or ignored, actual outcome, financial impact. 18+ months of this data is the most valuable dataset in UK pharmacy procurement intelligence. Cannot be purchased. Cannot be replicated without the customer relationships. This is the table that, if built Day 1, becomes the platform's most defensible asset.

Irreplicable after 18 months

Five Moves — What I'd Do in the Next 30 Days

Actionable Recommendations
01
This Week

Run the Backtest. Before Any Infrastructure.

Download NHSBSA Drug Tariff history 2014–2024 and all price concession records from nhsbsa.nhs.uk/open-data. Write a 100-line Python script. 02
Week 1

Build the Molecule Master Database First. Before Any Other Data Integration.

Create a clean mapping of all NHS-dispensed molecules: BNF code → drug name → form → strength → WaveData ID → OpenPrescribing ID → DaSH product name. This is the Rosetta Stone of the platform. Without it, every data JOIN from week 2 onwards is unreliable. Pull from NHSBSA Part VIIIA (the definitive source), enrich with OpenPrescribing BNF tree, and add a manual QA pass on the top 500 molecules by dispensing volume. Estimate: 3–4 days of data engineering.

03
Phase 1 Design

Redesign the Recommendation Output to Include Financial Impact.

Every recommendation must output: recommended action, weeks of stock suggested, current stock (pull from ERP/buying system or manual input), order quantity, cost to act today, expected saving if prediction correct, risk-adjusted net value, and decision deadline. The formula is simple: (saving × probability) − (overstocking_cost × (1−probability)). When positive: recommend action. When negative: monitor. This single change transforms the platform from "interesting dashboard" to "procurement decision engine." Design this financial calculator before the UI is built — it's far cheaper to add now than to retrofit.

04
Architecture Decision

Move Outcome Tracking From Phase 2 to Day 1 Infrastructure.

The recommendations table with outcome tracking fields (was_acted_on, actual_outcome, pnl_impact_gbp) must be part of the Phase 1 schema, even if it's only populated manually in Phase 1. The data you collect in the first 6 months of operation is irreplaceable. If you wait until Phase 2 to design this, you'll have 6 months of recommendation history with no recorded outcomes — and you'll need to reconstruct it manually. Build the table now. Populate it from Day 1, even with spreadsheet entries if needed.

05
Strategic

Treat WaveData as Optional Enhancement, Not Required Dependency. Build the Floor Calculator From NHSBSA Data First.

Prove the unprofitable floor can be calculated from freely available NHSBSA data alone. Use the backtest to measure its accuracy. If NHSBSA-derived floors predict >60% of concessions, you have a negotiating position with WaveData: "we can launch without you, but your data would improve our accuracy from 65% to 80% — here's what we'd pay for that." If you need WaveData before launch, they control your timeline, your cost structure, and your competitive positioning. Never let a single vendor control any of those three.
